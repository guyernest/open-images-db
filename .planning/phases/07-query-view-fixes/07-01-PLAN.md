From queries/audit/02-hierarchy-structure.sql (recursive CTE pattern for Athena):
```sql
-- Athena requires column aliases on recursive CTE definitions
WITH RECURSIVE roots(mid) AS (
  SELECT DISTINCT parent_mid AS mid
  FROM __DATABASE__.label_hierarchy
  WHERE NOT EXISTS (
    SELECT 1 FROM __DATABASE__.label_hierarchy c WHERE c.child_mid = parent_mid
  )
),
tree(mid, parent_mid, depth) AS (
  SELECT mid, CAST(NULL AS VARCHAR) AS parent_mid, 0 AS depth
  FROM roots
  UNION ALL
  SELECT h.child_mid AS mid, h.parent_mid AS parent_mid, t.depth + 1 AS depth
  FROM tree t
  JOIN __DATABASE__.label_hierarchy h ON t.mid = h.parent_mid
  WHERE t.depth < 20
)
SELECT t.mid, COALESCE(cd.display_name, t.mid) AS display_name, t.depth, ...
```

2. Create queries/views/06-hierarchy-relationships.sql:
   - Single CREATE OR REPLACE VIEW statement using __DATABASE__ placeholder
   - View name: hierarchy_relationships
   - Purpose: Like labeled_relationships but with ancestor class expansion
   - Structure: Two recursive CTEs that walk UP from each MID to find all ancestors
     - ancestors_1: walks label_hierarchy from label_name_1 upward, collecting ancestor MIDs + depth
     - ancestors_2: walks label_hierarchy from label_name_2 upward, collecting ancestor MIDs + depth
   - Include ALL columns from labeled_relationships (image_id, original_url, original_landing_url, license, author, original_size, thumbnail_300k_url, label_name_1, display_name_1, label_name_2, display_name_2, relationship_label, bbox coords)
   - PLUS: ancestor_name_1 (VARCHAR), ancestor_name_2 (VARCHAR), depth_1 (INT), depth_2 (INT)
   - depth_1/depth_2: 0 = the direct class itself, 1 = parent, 2 = grandparent, etc.
   - The view JOINs relationships to the ancestor CTEs, producing one row per (relationship, ancestor1, ancestor2) combination
   - This means a "Man on Horse" row also appears as "Person on Horse" (depth_1=1), "Person on Animal" (depth_1=1, depth_2=1), etc.
   - Use INNER JOIN to labeled_relationships (reuse existing view) then LEFT JOIN to ancestor CTEs
   - Actually, better approach: build ancestors inline rather than depending on the view, to avoid view-on-view issues in Athena. Use the same JOIN pattern as 04-labeled-relationships but add the ancestor CTEs.

   Implementation approach for hierarchy_relationships:
   ```sql
   WITH RECURSIVE
   -- Walk UP from any MID to all its ancestors
   ancestors(mid, ancestor_mid, depth) AS (
     -- Base: every MID is its own ancestor at depth 0
     SELECT DISTINCT label_name AS mid, label_name AS ancestor_mid, 0 AS depth
     FROM __DATABASE__.class_descriptions
     UNION ALL
     -- Recursive: walk up via label_hierarchy (child -> parent)
     SELECT a.mid, h.parent_mid AS ancestor_mid, a.depth + 1 AS depth
     FROM ancestors a
     JOIN __DATABASE__.label_hierarchy h ON a.ancestor_mid = h.child_mid
     WHERE a.depth < 20
   )
   SELECT
     r.image_id, i.original_url, i.original_landing_url, i.license, i.author,
     i.original_size, i.thumbnail_300k_url,
     r.label_name_1, cd1.display_name AS display_name_1,
     r.label_name_2, cd2.display_name AS display_name_2,
     r.relationship_label,
     r.x_min_1, r.x_max_1, r.y_min_1, r.y_max_1,
     r.x_min_2, r.x_max_2, r.y_min_2, r.y_max_2,
     acd1.display_name AS ancestor_name_1,
     acd2.display_name AS ancestor_name_2,
     a1.depth AS depth_1,
     a2.depth AS depth_2
   FROM __DATABASE__.relationships r
   JOIN __DATABASE__.images i ON r.image_id = i.image_id
   JOIN __DATABASE__.class_descriptions cd1 ON r.label_name_1 = cd1.label_name
   JOIN __DATABASE__.class_descriptions cd2 ON r.label_name_2 = cd2.label_name
   JOIN ancestors a1 ON r.label_name_1 = a1.mid
   JOIN ancestors a2 ON r.label_name_2 = a2.mid
   JOIN __DATABASE__.class_descriptions acd1 ON a1.ancestor_mid = acd1.label_name
   JOIN __DATABASE__.class_descriptions acd2 ON a2.ancestor_mid = acd2.label_name;
   ```

   NOTE: The ancestors CTE seeds from class_descriptions (all ~20,931 MIDs). This may be too expensive for a view. If so, consider seeding only from MIDs that appear in relationships:
   ```sql
   SELECT DISTINCT mid FROM (
     SELECT label_name_1 AS mid FROM __DATABASE__.relationships

   )
   ```
   Use this narrower seed to keep the recursive expansion manageable.
