  <action>
Create the queries/audit/ directory and four SQL files. All SQL must use __DATABASE__ placeholder (substituted at runtime like other SQL files in this project).

**01-relationship-types.sql** (AUDIT-01):
Query 1: Count distinct relationship types from the raw relationships table with per-type counts, ordered by count descending. Columns: relationship_label, instance_count.
Query 2: Same query but from the labeled_relationships view, to compare what's visible through the view vs raw table.
Query 3: Total row counts from both relationships table and labeled_relationships view (to quantify the INNER JOIN drop).
Separate queries with semicolons (runner splits on semicolons like create-tables.sh does).

**02-hierarchy-structure.sql** (AUDIT-02):
Query 1: Find root nodes -- parent_mid values in label_hierarchy that never appear as child_mid. Join class_descriptions to show display names. Columns: root_mid, root_display_name.
Query 2: Compute max depth using a recursive CTE. Start from root nodes, walk child edges, track depth. Return max_depth.
Query 3: Full hierarchy traversal -- recursive CTE that walks from roots to leaves, producing: mid, display_name, depth, parent_mid, parent_display_name. Order by depth, display_name. This gives the full tree structure.
Query 4: Branch density -- for each node with children, count the number of direct children. Join class_descriptions for names. Columns: parent_mid, parent_name, child_count. Order by child_count desc. Top-heavy branches = rich, single-child = sparse.
Query 5: Coverage analysis -- count distinct MIDs in label_hierarchy vs class_descriptions. Show the gap (~600 vs ~20K).
Note: The hierarchy uses MID format (parent_mid, child_mid), NOT label_name. Join to class_descriptions ON label_hierarchy.parent_mid = class_descriptions.label_name (both are MID format like /m/01g317).

**03-entity-pair-relationships.sql** (AUDIT-03):
Query 1: From the relationships table, group by (label_name_1, label_name_2, relationship_label), join class_descriptions twice to get display names. Columns: display_name_1, display_name_2, relationship_label, instance_count. Order by instance_count desc. This answers "which entity pairs participate in which relationship types."
Query 2: Specifically search for Person-Horse relationships (or any /m/ codes that map to Person and Horse) to answer the user's "people on horses" question. Use display_name LIKE '%Person%' and '%Horse%' patterns through class_descriptions joins.

**04-dropped-rows-analysis.sql**:
Query 1: Find the ~886 rows in relationships that are NOT in labeled_relationships. Use LEFT JOIN from relationships to class_descriptions on label_name_1 and label_name_2, filter WHERE cd1.label_name IS NULL OR cd2.label_name IS NULL. Show: relationship_label, label_name_1, label_name_2, count. This traces which relationship types and MIDs are lost.
Query 2: List the distinct MIDs from relationships that have no match in class_descriptions. Columns: orphan_mid, side (1 or 2), occurrence_count.

Each SQL file should have a comment header following the project pattern:
