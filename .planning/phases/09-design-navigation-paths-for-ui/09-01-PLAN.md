  <action>
Create `design/tool-definitions/tools.json` containing the complete MCP tool definitions array. Define these tools:

1. **find_images** -- Primary search entry point
   - inputSchema: query (string, required), limit (number, default 20), relationship (string, optional), min_confidence (number, optional)
   - outputSchema: Defines the three-layer response structure:
     - structuredContent: query echo, total_results count, page number, facets object (categories array from class_hierarchy children, relationships array excluding "is" by default, confidence_ranges), summary text
     - content: Conversation text describing results with narrowing suggestions
     - _meta: images array (id, thumbnail_300k_url, labels array, confidence, has_boxes boolean, has_relationships boolean), hierarchy_context (path from class_hierarchy root_path, children array, depth), query_metadata (sql_views_used, execution_time_ms)
   - _meta.ui: resourceUri "ui://widgets/results-grid.html", visibility ["model", "app"]

2. **narrow_results** -- Refinement within current result set
   - inputSchema: previous_query (string, required), filter (string, required -- supports "category:Poodle", "relationship:ride", "confidence:>0.8", free text), page (number, default 1), limit (number, default 20)
   - outputSchema: Same structure as find_images (widget re-renders in place)
   - _meta.ui: visibility ["app"] only (widget calls this directly, model does not)

3. **get_image_details** -- Single image deep view
   - inputSchema: image_id (string, required)
   - outputSchema:
     - structuredContent: image_id, thumbnail_300k_url, original_url, metadata (author, license, title, rotation, original_size)
     - content: Natural language description of image contents
     - _meta: labels array (display_name, source, confidence), boxes array (display_name, coordinates x_min/y_min/x_max/y_max, confidence, is_occluded, is_truncated, is_group_of, is_depiction), relationships array (display_name_1, relationship_label, display_name_2, subject_box, object_box), masks array (display_name, mask_path, predicted_iou), navigate_actions (more_like_this, same_objects, same_relationships, same_category)
   - _meta.ui: resourceUri "ui://widgets/image-detail.html", visibility ["model", "app"]

4. **explore_category** -- Hierarchy browsing
   - inputSchema: class_name (string, required), depth (number, default 2 -- how many levels down to show), include_samples (boolean, default true)
   - outputSchema:
     - structuredContent: class_name, root_path (from class_hierarchy), depth, child_count, edge_type
     - content: Description of category with available subcategories and relationships
     - _meta: hierarchy_tree (recursive structure: name, mid, depth, edge_type, is_leaf, children array, sample_count), relationship_summary (array of relationship_label + count, excluding "is"), sample_images (array of id + thumbnail_300k_url, 3-5 per subcategory if include_samples)
   - _meta.ui: resourceUri "ui://widgets/hierarchy-browser.html", visibility ["model", "app"]

5. **start_code_mode** -- MCP prompt (not a tool, define as prompt)
   - Define as MCP prompt, not tool
   - Loads 00-mcp-reference.sql as resource context
   - Includes code generation instructions: use Athena/Trino dialect, replace __DATABASE__ with open_images, prefer views over base tables, warn about pitfalls (MIDs vs display_name, "is" dominance, leaf-only relationships)
   - Returns conversation text only (no widget)

Format the file as a JSON object with two top-level keys: "tools" (array of tool definitions) and "prompts" (array of prompt definitions). Each tool follows MCP tool schema: name, description, inputSchema (JSON Schema), outputSchema (JSON Schema with structuredContent/content/_meta), _meta.ui.

Include descriptive comments as "description" fields at every level so the LLM understands the purpose of each field. Tool descriptions should be specific enough that the LLM picks the right tool without ambiguity:
- find_images: "Search for images by class name, object type, or natural language description. Use this for initial queries. Returns a visual grid."
- narrow_results: "Refine the current search results by adding filters. Called by the widget directly -- do not call this from conversation."
- get_image_details: "Get full details for a single image including all annotations, bounding boxes, relationships, and metadata."
- explore_category: "Browse the class hierarchy tree starting from a category. Shows subcategories, sample images, and available relationships."

<task type="auto">
  <name>Task 2: Create interaction model decision framework</name>
  <files>design/patterns/interaction-model.md</files>
  <action>
Create `design/patterns/interaction-model.md` documenting the interaction mechanisms and when to use each.

Structure the document as:

**1. Three Interaction Mechanisms**
For each mechanism (tools/call, ui/message, ui/update-model-context), document:
- What it does (technical behavior)
- Who initiates it (widget, model, or user)
- What happens in the conversation (silent vs new turn)
- Reliability assessment (based on RESEARCH.md findings)
- Code pattern (JavaScript snippet showing usage)

**2. Decision Tree**
Create a decision tree (as nested bullet list with clear if/then structure):
- Is the user staying within the same search context? -> tools/call
  - Facet click -> tools/call narrow_results
  - Pagination -> tools/call narrow_results with page param
  - Sort change -> tools/call narrow_results
  - Hierarchy node expand -> tools/call explore_category
- Is the user switching to a fundamentally different view? -> ui/message
  - Click image thumbnail for detail -> ui/message (triggers get_image_details, new widget)
  - "Show me more like this" -> ui/message (triggers find_images with new context)
  - "What else is in this scene?" -> ui/message (triggers find_images with relationship filter)
- Does the model need to know what the user is looking at? -> ui/update-model-context
  - User scrolls to different section of results
  - User toggles annotation layers in detail view
  - User expands hierarchy branch

**3. Fallback Patterns**
For each ui/message action, document the fallback behavior when the LLM does not trigger the expected tool:
- Widget should display a text prompt the user can copy/paste into conversation
- Widget should show a "Try typing: [suggested text]" hint after 5 seconds if no response
- Design the ui/message text to be highly specific so the LLM is more likely to call the right tool (e.g., include tool name hint: "I want to see details for image 000a1249af2bc5f0 [get_image_details]")

**4. Error Handling Patterns**
- tools/call timeout: Show spinner for 3s, then "Loading..." text, then error state with retry button after 10s
- tools/call returns empty results: Widget shows "No results" state with suggestions (broaden query, try different category)
- ui/message no response: Widget shows fallback text prompt
- Network error: Widget shows offline state, does NOT retry automatically

**5. Data Flow Diagrams**
For each primary interaction, show the data flow as ASCII sequence diagram:
- Facet click: User -> Widget -> tools/call(narrow_results) -> MCP Server -> Athena -> response -> Widget re-render
- Thumbnail click: User -> Widget -> ui/message -> LLM -> tools/call(get_image_details) -> MCP Server -> new Widget
- Hierarchy expand: User -> Widget -> tools/call(explore_category) -> MCP Server -> Widget re-render

**6. Visibility Matrix**
Table showing each tool's visibility setting and rationale:
| Tool | Model | App (Widget) | Rationale |
|------|-------|-------------|-----------|
| find_images | yes | yes | Model calls on user query, widget calls for "more like this" |
| narrow_results | no | yes | Widget-only for fast filtering, model should use find_images for new searches |
| get_image_details | yes | yes | Model calls from conversation, widget calls from thumbnail click |
| explore_category | yes | yes | Model calls for hierarchy questions, widget calls for tree expand |
