  <action>
Create `design/widget-specs/results-grid.md` specifying the primary search results widget. This is the most complex widget -- it contains both the image grid and inline refinement facets.

Structure the spec as:

**1. Overview**
- Purpose: Display search results as a visual thumbnail grid with inline filtering
- Rendered by: find_images and narrow_results tool responses
- Self-contained: facets, pagination, and grid all live in this single iframe
- Data source: _meta.images array from tool response, structuredContent.facets for filter options

**2. Layout Specification**
Design a responsive layout (the iframe will be roughly 600-800px wide in ChatGPT):
- **Top bar**: Query summary text (from structuredContent.summary), result count
- **Facet row** (below top bar, horizontal scroll): Pill-shaped filter chips for categories (from facets.categories) and relationships (from facets.relationships, excluding "is"). Active filters shown as filled pills, inactive as outlined. Clicking a pill triggers tools/call to narrow_results.
- **Image grid**: CSS Grid, 3 columns on wide viewport, 2 on narrow. Each cell shows:
  - thumbnail_300k_url as img element
  - Overlay bar at bottom with primary label (first item in labels array) and confidence badge
  - Hover: show all labels as tooltip
  - Click: triggers ui/message to request get_image_details for that image_id
- **Pagination bar** (below grid): "Load more" button (not page numbers -- infinite scroll feel). Triggers tools/call to narrow_results with page+1.
- **Empty state**: "No images match your filters. Try broadening your search." with a "Clear all filters" button.

**3. Data Contract**
Document the exact JSON shape this widget expects from _meta:
```
_meta.images[]: { id, thumbnail_300k_url, labels[], confidence, has_boxes, has_relationships }
structuredContent.facets: { categories[], relationships[], confidence_ranges[] }
structuredContent.total_results: number
structuredContent.page: number
structuredContent.query: string
structuredContent.summary: string
```

**4. Interaction Behaviors**
For each user action, specify mechanism (from interaction-model.md) and parameters:
| Action | Mechanism | Tool | Parameters | Widget Behavior |
|--------|-----------|------|------------|-----------------|
| Click facet pill | tools/call | narrow_results | filter: "category:{name}", previous_query | Re-render grid with filtered results, update active facet state |
| Click relationship pill | tools/call | narrow_results | filter: "relationship:{label}", previous_query | Same as above |
| Click "Load more" | tools/call | narrow_results | page: current+1, previous_query, active filters | Append new images to existing grid |
| Click image thumbnail | ui/message | (triggers get_image_details) | "Show details for image {id}" | Show loading overlay on clicked image. Fallback: display "Try typing: show details for image {id}" |
| Click "Clear all filters" | tools/call | find_images | original query, no filters | Reset grid to unfiltered state |

**5. State Management**
- Widget maintains: current query, active filters array, current page, loaded images array
- On tools/call response: merge or replace images depending on action (paginate = append, filter = replace)
- On initial render: parse _meta from tool response, render grid, populate facet pills from structuredContent.facets

**6. Accessibility**
- Image alt text: concatenation of labels array (e.g., "Dog, Poodle, Outdoors")
- Facet pills: role="button", aria-pressed for active state
- Grid: role="grid" with role="gridcell" per image
- Keyboard: Tab through facets, Enter to toggle. Tab into grid, arrow keys to navigate, Enter to select image.

**7. Responsive Behavior**
- 600px+: 3-column grid, horizontal facet row
- 400-599px: 2-column grid, facet row wraps
- Below 400px: 2-column grid, facet row scrolls horizontally

Create two widget specification documents:

**A. `design/widget-specs/image-detail.md`** -- Single image deep view

**1. Overview**
- Purpose: Display a single image with all annotations, overlays, and navigation actions
- Rendered by: get_image_details tool response
- Shows the image at maximum available size with toggleable annotation layers

**2. Layout Specification**
Two-panel layout within the iframe:
- **Image panel** (top 60% of iframe height):
  - Image displayed at full width (using original_url or thumbnail_300k_url)
  - Annotation overlay layers (rendered as absolutely-positioned divs over the image):
    - Bounding boxes: colored rectangles using x_min/y_min/x_max/y_max (normalized to image dimensions). Label text above each box. Color-coded by class.
    - Relationship lines: dashed lines connecting subject box center to object box center, with relationship_label at midpoint
  - Layer toggle buttons (top-right of image): "Boxes" toggle, "Labels" toggle, "Relationships" toggle. All on by default.
  - If no boxes/relationships: show labels as a chip bar below the image instead

- **Info panel** (bottom 40%, scrollable):
  - Metadata section: author, license (as link), title, original_size formatted, rotation
  - Labels section: Table of display_name, source (human/machine badge), confidence (percentage bar)
  - Relationships section: List of "display_name_1 [relationship_label] display_name_2" with confidence. Filter out "is" relationships by default, toggle to show.
  - Navigate-from-image actions (as button row at bottom):
    - "More with [primary label]" -> ui/message: "Find more images of {primary_label}"
    - "Similar scenes" -> ui/message: "Find images with similar objects to image {id}" (uses the image's label set)
    - "Explore [category]" -> ui/message: "Explore the {root_category} category" (uses hierarchy path)

**3. Data Contract**
```
_meta.labels[]: { display_name, source, confidence }
_meta.boxes[]: { display_name, x_min, y_min, x_max, y_max, confidence, is_occluded, is_truncated, is_group_of, is_depiction }
_meta.relationships[]: { display_name_1, relationship_label, display_name_2, subject_box: {x_min,y_min,x_max,y_max}, object_box: {x_min,y_min,x_max,y_max} }
_meta.masks[]: { display_name, mask_path, predicted_iou }
_meta.navigate_actions: { more_like_this: string, same_objects: string[], same_relationships: string[], same_category: string }
structuredContent: { image_id, thumbnail_300k_url, original_url, metadata: { author, license, title, rotation, original_size } }
```

**4. Interaction Behaviors**
| Action | Mechanism | Behavior |
|--------|-----------|----------|
| Toggle annotation layer | Local state | Show/hide overlay divs |
| Click navigate action | ui/message | Send pre-formatted message. Show fallback text after 5s |
| Click label name | ui/message | "Find more images of {label}" |
| Click relationship | ui/message | "Find images where {subject} {relationship} {object}" |
| Hover bounding box | Local state | Highlight box, show label tooltip with confidence |

**5. Annotation Rendering Rules**
- Normalize coordinates: x_min * image_width, y_min * image_height, etc.
- Box colors: use a deterministic color palette based on display_name hash (so same class always gets same color)
- Relationship lines: draw from center of subject_box to center of object_box, label at midpoint
- Overlapping boxes: most confident on top (higher z-index)
- is_depiction boxes: render with dashed border style
- is_group_of boxes: render with double border style

**B. `design/widget-specs/hierarchy-browser.md`** -- Category tree exploration

**1. Overview**
- Purpose: Browse the class hierarchy as an interactive tree, with sample images at each level
- Rendered by: explore_category tool response
- Transitions between tree-view (browsing hierarchy) and grid-view (viewing category images) within the same iframe

**2. Layout Specification**
Two modes within one widget:

**Tree Mode (default):**
- **Breadcrumb trail** (top): Shows path from root to current node (e.g., Entity > Animal > Carnivore > Dog). Each segment is clickable to navigate up. Uses root_path from class_hierarchy view.
- **Current node header**: Class name, total image count, relationship summary (e.g., "12 relationship types, most common: ride (45), on (32)")
- **Children list**: Expandable items, each showing:
  - Class name + image count badge
  - Edge type indicator ("subcategory" vs "part" -- use different icon)
  - 3 sample thumbnails inline (if include_samples was true)
  - Expand arrow if node has children (not is_leaf)
  - Click class name: triggers tools/call to explore_category for that class
  - Click sample thumbnail: triggers ui/message for get_image_details
- **"View all images" button**: Transitions to grid mode for current category

**Grid Mode (after "View all images"):**
- Same layout as results-grid widget but within this iframe
- Back button to return to tree mode
- Facets limited to subcategories of current class
- Triggers tools/call to find_images with category filter

**3. Data Contract**
```
_meta.hierarchy_tree: recursive { name, mid, depth, edge_type, is_leaf, children[], sample_count }
_meta.relationship_summary[]: { relationship_label, count }
_meta.sample_images[]: { id, thumbnail_300k_url, class_name }
structuredContent: { class_name, root_path, depth, child_count, edge_type }
```

**4. Interaction Behaviors**
| Action | Mechanism | Tool | Behavior |
|--------|-----------|------|----------|
| Click child class name | tools/call | explore_category | Re-render tree with clicked class as root, update breadcrumb |
| Click breadcrumb segment | tools/call | explore_category | Navigate up to that level |
| Expand/collapse node | Local state | none | Toggle children visibility (data already loaded for current depth) |
| Click sample thumbnail | ui/message | (get_image_details) | "Show details for image {id}". Fallback text. |
| Click "View all images" | tools/call | find_images | Transition to grid mode with category filter |
| Click back (grid mode) | Local state | none | Return to tree mode (data still in memory) |

**5. Tree Rendering Rules**
- Initial render shows 2 levels deep (from explore_category depth parameter)
- Deeper levels loaded on demand via tools/call
- Leaf nodes: no expand arrow, just class name + sample thumbnails
- "part" edge_type nodes: shown with a different visual treatment (italicized or indented with "part of" label) to distinguish from subcategory nodes
- Maximum visible depth in one render: 3 levels (to avoid overwhelming the UI)
