  <action>
Create scripts/run-validation.sh following the established runner pattern from run-audit.sh. Source common.sh and athena.sh. Support --dry-run and --help flags.

The script performs 4 categories of validation checks, plus round-trip traces:

**Category 1: Non-empty results**
For each SQL file in queries/examples/ (01-08, skip 00-mcp-reference.sql which is documentation only):
- Use process_sql_file to execute each file against Athena
- After each query execution, use `aws athena get-query-results` with ATHENA_LAST_QUERY_ID to check row count
- PASS if result has >= 1 data row (after header), FAIL if 0 rows
- Track results in arrays for summary

**Category 2: Human-readable names**
Run targeted checks that display_name columns contain real names, not raw MIDs:
- Check 1: `SELECT display_name_1 FROM open_images.labeled_relationships LIMIT 1` — result should NOT start with /m/
- Check 2: `SELECT display_name FROM open_images.class_hierarchy WHERE depth = 1 LIMIT 1` — result should NOT start with /m/
- Check 3: `SELECT ancestor_name_1 FROM open_images.hierarchy_relationships LIMIT 1` — result should NOT start with /m/

**Category 3: Row count sanity**
Compare live counts against audit report ranges (not exact — use 10% tolerance):
- Relationship types: `SELECT COUNT(DISTINCT relationship_label) FROM open_images.relationships` — expect 27
- Total relationship rows: `SELECT COUNT(*) FROM open_images.relationships` — expect ~27,243
- Hierarchy MIDs: `SELECT COUNT(DISTINCT mid) FROM open_images.class_hierarchy` — expect ~602
- View relationships: `SELECT COUNT(*) FROM open_images.labeled_relationships` — expect ~26,357

**Category 4: Cross-view consistency**
- Query `Person` children from class_hierarchy (expect Man, Woman, Boy, Girl)
- Query hierarchy_relationships for ancestor_name_1 = 'Person' and verify display_name_1 values are subset of Person's descendants

**Round-trip traces (3 traces):**

Trace 1 — Man on Horse:
- Layer 1: `SELECT COUNT(*) FROM relationships WHERE label_name_1 = '/m/04yx4' AND label_name_2 = '/m/03k3r' AND relationship_label = 'on'` (raw MIDs for Man, Horse)
- Layer 2: `SELECT COUNT(*) FROM labeled_relationships WHERE display_name_1 = 'Man' AND display_name_2 = 'Horse' AND relationship_label = 'on'`
- Layer 3: `SELECT COUNT(*) FROM hierarchy_relationships WHERE display_name_1 = 'Man' AND display_name_2 = 'Horse' AND relationship_label = 'on' AND depth_1 = 0 AND depth_2 = 0`
- Verify: Layer 1 count = Layer 2 count = Layer 3 count (same data, different access paths)

Trace 2 — Animal subtree:
- `SELECT display_name FROM class_hierarchy WHERE root_path LIKE 'Entity > Animal%' ORDER BY display_name`
- Verify Dog, Cat, Horse all appear in results

Trace 3 — Woman wears Hat:
- Same 3-layer pattern as Trace 1 but with Woman (/m/03bt1vf) + Hat (/m/02dl1y) + 'wears'
- Verify counts match across layers

**Output format:**
Print a summary table at the end:
```
============================================
Validation Summary
============================================
Category               | Passed | Failed | Total
Non-empty results      |     N  |     N  |     N
Human-readable names   |     N  |     N  |     N
Row count sanity       |     N  |     N  |     N
Cross-view consistency |     N  |     N  |     N
Round-trip traces      |     N  |     N  |     N
--------------------------------------------
TOTAL                  |     N  |     N  |     N

OVERALL: PASS/FAIL
```

Exit 0 if all pass, exit 1 if any fail. Use continue-on-failure pattern (don't exit on first failure).

Note: For dry-run mode, skip actual Athena execution and just log what would run. The MID values for Man (/m/04yx4), Horse (/m/03k3r), Woman (/m/03bt1vf), Hat (/m/02dl1y) are from the class_descriptions table — verify these are correct by checking the audit report or using athena_query_scalar to look them up at runtime rather than hardcoding (more robust). Actually, hardcoding is fine for validation — these are stable identifiers in Open Images V7.
